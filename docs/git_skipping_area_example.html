Skipping the Staging Area

Scenario: Removing Temporary Files

You've been working on a project and created several temporary files for testing. Now you want to clean up your repository by removing these files before committing your final changes.

1
Identify temporary files
List all temporary files in your project directory
2
Remove files using git rm
Use the git rm command to delete files and stage the deletion in one step
git rm temp_file1.txt temp_file2.txt temp_file3.txt
3
Commit the changes
The deletions are already staged, so you can commit directly
git commit -m "Remove temporary files"
Before
main.js
temp_file1.txt
temp_file2.txt
temp_file3.txt
After
main.js
Key Point
Using git rm combines two operations (deleting files and staging the deletion) into one command, saving you from having to separately stage the changes with git add.

Scenario: Reorganizing Project Structure

You're working on a project and realize that several JavaScript files would be better organized in a dedicated 'js' folder. You want to move these files to maintain a cleaner project structure.

1
Create the new directory
Create the 'js' directory where you'll move the files
mkdir js
2
Move files using git mv
Use the git mv command to move files and stage the changes in one step
git mv app.js js/
git mv utils.js js/
3
Commit the changes
The file movements are already staged, so you can commit directly
git commit -m "Reorganize JavaScript files into dedicated folder"
Before
index.html
app.js
utils.js
styles.css
After
index.html
js/
app.js
utils.js
styles.css
Key Point
Using git mv preserves the file's history, unlike using the system's mv command followed by git add. This is important for maintaining a complete project history.

Undoing Changes

Scenario: Reverting an Incorrect Commit

You've just committed a change that introduced a bug to your application. You need to revert this commit to restore the previous working state of the code.

1
Identify the problematic commit
Use git log to find the commit hash of the commit you want to revert
git log --oneline
a1b2c3d (HEAD -> main) Add new feature that breaks existing functionality
e4f5g6h Fix login bug
i7j8k9l Update documentation
2
Revert the commit
Use git revert with the commit hash to create a new commit that undoes the changes
git revert a1b2c3d
3
Verify the reversion
Check that the changes have been properly reverted
git log --oneline
m9n8o7p (HEAD -> main) Revert "Add new feature that breaks existing functionality"
a1b2c3d Add new feature that breaks existing functionality
e4f5g6h Fix login bug
i7j8k9l Update documentation
Key Point
git revert creates a new commit that undoes the changes from a previous commit. This is different from git reset, which removes commits entirely from the history. Revert is safer for shared repositories.

Scenario: Fixing a Commit Message

You've just committed a change but noticed a typo in the commit message. You want to fix the message without changing any of the actual code changes.

1
Check the current commit
Verify the commit message that needs to be fixed
git log --oneline -1
a1b2c3d (HEAD -> main) Addd new user authentication feature
2
Amend the commit message
Use git commit --amend to fix the commit message
git commit --amend -m "Add new user authentication feature"
3
Verify the fix
Check that the commit message has been corrected
git log --oneline -1
f4g5h6i (HEAD -> main) Add new user authentication feature
Key Point
git commit --amend allows you to modify the most recent commit. It's useful for fixing typos in commit messages or adding forgotten changes to the last commit. However, don't amend commits that have already been pushed to a shared repository.

Branching and Merging Concepts

Scenario: Feature Development Workflow

You're working on a team developing a web application. You need to implement a new feature while other team members continue working on the main codebase. You want to keep your work isolated until it's ready for integration.

1
Create a feature branch
Create a new branch for your feature development
git checkout -b feature/user-profile main
2
Develop the feature
Make changes and commit them to your feature branch
git add .
git commit -m "Add user profile page structure"
git add .
git commit -m "Implement user profile functionality"
3
Switch back to main branch
Return to the main branch to integrate your feature
git checkout main
4
Merge the feature branch
Merge your feature branch into the main branch
git merge feature/user-profile
5
Clean up
Delete the feature branch after merging
git branch -d feature/user-profile
Key Point
Branching allows you to isolate your work from the main codebase. This is especially important in team environments where multiple developers are working on different features simultaneously.

Scenario: Hotfix for Production Issue

Your team has discovered a critical bug in the production version of your application. You need to fix it quickly while other development continues on the main branch.

1
Create a hotfix branch
Create a branch from the production tag or commit
git checkout -b hotfix/critical-bug v1.0
2
Implement the fix
Make the necessary changes to fix the bug
git add .
git commit -m "Fix critical security vulnerability"
3
Tag the fixed version
Create a new tag for the fixed version
git tag -a v1.0.1 -m "Critical bug fix"
4
Merge to main branch
Switch to main and merge the hotfix
git checkout main
git merge hotfix/critical-bug
5
Clean up
Delete the hotfix branch
git branch -d hotfix/critical-bug
Key Point
Hotfix branches are short-lived branches created directly from production tags. They allow you to fix critical issues without disrupting ongoing development work on other branches.

Using Merging to Combine Data

Scenario: Integrating Team Work

Your team has been working on different features in separate branches. Now it's time to integrate everyone's work into the main branch for the next release.

1
Switch to main branch
Ensure you're on the main branch where you'll merge all features
git checkout main
2
Update main branch
Pull the latest changes to ensure your main branch is up to date
git pull origin main
3
Merge first feature branch
Merge the first feature branch into main
git merge feature/user-authentication
4
Merge remaining feature branches
Continue merging other feature branches one by one
git merge feature/payment-processing
git merge feature/reporting-dashboard
5
Resolve any conflicts
If conflicts occur during merging, resolve them and continue
git add .
git commit
6
Push the integrated code
Push the merged main branch to the remote repository
git push origin main
Key Point
When merging multiple branches, it's often best to merge one branch at a time and resolve any conflicts immediately. This makes the process more manageable than trying to merge multiple branches simultaneously.

Scenario: Merging with Rebase for Clean History

You want to integrate your feature branch into the main branch, but you prefer to maintain a clean, linear project history without merge commits.

1
Switch to your feature branch
Ensure you're on the branch you want to integrate
git checkout feature/user-profile
2
Update the feature branch
Fetch the latest changes from the main branch
git fetch origin
3
Rebase onto main
Rebase your feature branch onto the latest main branch
git rebase origin/main
4
Switch to main branch
Move to the main branch for the final integration
git checkout main
5
Fast-forward merge
Perform a fast-forward merge since the branches are now aligned
git merge feature/user-profile
6
Clean up
Delete the feature branch after successful integration
git branch -d feature/user-profile
Key Point
Rebase rewrites your branch's history by moving its base to a newer commit. This creates a linear history without merge commits, which can be easier to read and understand. However, never rebase commits that have already been shared with others.

Managing Merge Conflicts

Scenario: Conflicting Changes in the Same File

You and a teammate have both made changes to the same file in different branches. When you try to merge your branches, Git reports a conflict that needs to be resolved manually.

1
Attempt the merge
Try to merge the branches, which will result in a conflict
git merge feature/user-profile
Auto-merging app.js
CONFLICT (content): Merge conflict in app.js
Automatic merge failed; fix conflicts and then commit the result.
2
Check the status
Use git status to see which files have conflicts
git status
On branch main
You have unmerged paths.
(fix conflicts and run "git commit")
(use "git merge --abort" to abort the merge)

Changes to be committed:

modified: styles.css

Unmerged paths:
(use "git add ..." to mark resolution)

both modified: app.js
3
Open the conflicted file
Edit the file to resolve the conflicts
function updateUser() {
<<<<<<< HEAD
// Current branch implementation
const user = getUserById(id);
user.name = newName;
saveUser(user);
=======
// Feature branch implementation
const user = await fetchUser(id);
user.updateProfile({name: newName});
await user.save();
>>>>>>> feature/user-profile
}
4
Resolve the conflict
Edit the file to keep the desired changes and remove conflict markers
function updateUser() {
// Resolved implementation combining both approaches
const user = await fetchUser(id);
user.name = newName;
await user.save();
}
5
Stage the resolved file
Mark the conflict as resolved by staging the file
git add app.js
6
Complete the merge
Commit to finalize the merge
git commit -m "Merge feature/user-profile with conflict resolution"
Key Point
When resolving conflicts, Git marks the conflicted areas with <<<<<<<, =======, and >>>>>> markers. You need to manually edit the file to keep the desired changes and remove these markers before staging the file.
